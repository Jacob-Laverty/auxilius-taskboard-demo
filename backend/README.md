# Backend

Express + TypeScript REST API with Socket.IO for real-time broadcasting. Persists data to PostgreSQL via parameterized queries and manages schema with versioned migrations.

## Tech Stack

| | Choice | Why |
|---|---|---|
| Framework | Express 5 | Lightweight |
| Language | TypeScript | Type safety...shared type definitions with the frontend |
| Database | PostgreSQL | Required by spec; relational db also just makes sense here |
| Real-time | Socket.IO | Handles WebSockets...pairs with the frontend client |
| Migrations | node-pg-migrate | Versioned, reversible schema changes checked into source control. Even for a demo..would never let someone manage a DB without migrations... |
| Testing | Vitest + supertest | Vitest for test runner; supertest for in-process HTTP assertions without a live server |

## Architecture

**`createApp` factory.** The Express app is built by a factory function rather than a module-level singleton. `index.ts` owns the HTTP server and Socket.IO wiring; tests import `createApp` directly and inject a mock pool and broadcaster which means everything below index such as `app.ts` never has to be aware of anything in the network.

**`Broadcaster` interface.** Route handlers emit real-time events through a `Broadcaster` abstraction (`{ emit(event, payload) }`). In production this is backed by the Socket.IO instance...in tests this is mocked.

**Router → Repository → Validation.**
- `tasks.router.ts` — HTTP request/response handling and socket events
- `tasks.repository.ts` — all SQL queries; returns typed domain objects. Could start breaking this file down on excessive growth. For demo its fine for now. This would be like...priority number one to change for me...these queries need to be templated in a proper templating engine....having heredocs with queries in them gives me nightmares. **Semi Important:** Also also this whole file needs to be a PSQL ORM library for Typescript like TypeORM that a real app should be leveraging to serve as the middleman between the DB and the backend. There's really no universe I'd let something like the current implementation fly for anything other than an app that was created for a demo in the span of an hour or two.
- `tasks.validation.ts` — parses and validates raw request bodies; throws `ValidationError` on bad input. With more time this should align with an API spec. Its been a minute but I bet there's something out there that will auto-generate runtime validations based on an OpenApi spec.

**`wrap` async error handler.** Super small but super necessary middleware for the whole backend...just catches any thrown error from an async handler and routes it to a 400 (for `ValidationError`) or 500 response. Keeps individual route handlers free of try/catch which lets each individual route just Throw whatever it needs to throw without worry that it'll bubble an error to the top of the backend server stack. This would normally live in a middleware.ts file and would grow to handle all sorts of business logic error use cases as the app grows in spec.

## Database Schema

Managed by migrations in `migrations/`:

**`tasks` table**

| Column | Type | Notes |
|---|---|---|
| `id` | `uuid` | Primary key, generated by `gen_random_uuid()` |
| `task_number` | `bigint` | Sequential identity column; displayed in the UI as `TASK-N` |
| `title` | `text` | Required |
| `description` | `text` | Optional, nullable |
| `status` | `task_status` enum | `todo` \| `in_progress` \| `done`; defaults to `todo` |
| `created_at` | `timestamptz` | Set on insert |
| `updated_at` | `timestamptz` | Updated on every write |

**Why UUID + task_number?** The UUID is the stable primary key used in API paths and foreign key relationships. The sequential `task_number` (a Postgres identity column) gives a human-readable, sortable identifier for display (`TASK-1`, `TASK-2`). Keeping them separate means the legible ID stays compact and monotonic without making UUID the sort key.

**Indexes**

| Name | Column(s) | Type | Reason |
|---|---|---|---|
| `tasks_pkey` | `id` | Primary key, Unique B-Tree | Row lookups by ID in update and delete paths |
| `tasks_status_index` | `status` | B-tree | Loading tasks grouped by column; anticipated to be the dominant read pattern |
| `tasks_task_number_unique` | `task_number` | Unique B-tree | Enforces uniqueness of the human-readable ID; also supports `ORDER BY task_number` |

## Project Structure

```
src/
  index.ts             — server entry point; HTTP + Socket.IO wiring
  app.ts               — Express app factory; Broadcaster interface
  db.ts                — lazy-initialized pg.Pool singleton
  tasks.router.ts      — route definitions and event emission
  tasks.repository.ts  — SQL queries
  tasks.validation.ts  — request body parsing and validation
  types.ts             — shared domain types and socket event constants
  __tests__/
    tasks.test.ts      — full CRUD coverage with injected mocks
    healthcheck.test.ts
migrations/
  *_initial-schema.ts  — tasks table and task_status enum
  *_add-task-number.ts — task_number identity column
```

## Trade-offs & Known Limitations

**CORS is wide open.** Both Express (`cors()`) and Socket.IO (`origin: '*'`) accept requests from any origin. Appropriate for a demo; a production deployment would restrict this to the known frontend origin.

**No authentication on API endpoints.** The username is stored in `localStorage` on the client and never sent to the server. Any request from any client can create, update, or delete any task. Explicitly out of scope per the assignment, but worth noting as a known gap.

**Global pool singleton.** `db.ts` uses a module-level variable for the pool. This is noted in the code — the cleaner approach is a proper singleton class with a private constructor, but it's more overhead than this demo warrants.

**No rate limiting.** Mutation endpoints have no throttling. `express-rate-limit` would be the straightforward addition.

**No ORM** No robust resource mapping from the DB to our runtime objects. We just load an entire record from the DB after any action to ensure its injected into our object definition. Tightly coupled, not robust...leads to just raw queries written into heredocs.

## Environment Variables

| Variable | Description |
|---|---|
| `DATABASE_URL` | PostgreSQL connection string |

Copy `.env.example` from the project root to `backend/.env` for local development.

## Running Locally

See the root README for full local dev setup. Once the database is running and migrated:

```bash
cd backend
npm install
npm run dev    # tsx watch, restarts on file changes
```

## Tests

Tests use injected mocks — no database required.

```bash
npm test            # run once
npm run test:watch  # watch mode
```
